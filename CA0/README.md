## **PROJECT OVERVIEW**

This project implements a comprehensive **Kafka-based data pipeline** demonstrating real-time IoT data streaming using AWS cloud infrastructure. The system simulates an environmental monitoring scenario where sensor devices generate continuous data streams that are processed, transformed, and stored for analysis.

### **Architecture Summary:**
- **4 AWS EC2 VMs** (t2.micro, Free Tier optimized) running Ubuntu 22.04 LTS
- **Message Broker**: Apache Kafka 7.4.0 with ZooKeeper coordination
- **Database**: MongoDB 6.0 with web management interface
- **Data Flow**: Producer → Kafka → Consumer/Processor → MongoDB
- **Security**: VPC isolation, SSH key authentication, firewall rules
- **Management**: Web-based UIs for monitoring and administration

### **Data Pipeline Flow:**
1. **Python Producer** generates simulated sensor data (temperature, humidity, pressure)
2. **Kafka Broker** receives and partitions messages across 3 partitions  
3. **Python Consumer** processes messages and enriches with metadata
4. **MongoDB** stores processed data in document collections
5. **Management UIs** provide real-time monitoring and data visualization

### **Important Notes:**

**Simulation Environment**: This implementation uses **synthetic sensor data** generated by Python scripts to demonstrate the data pipeline functionality. No actual IoT sensors are connected to this system.

**Academic Project**: This CA0 assignment serves as the foundation for a progressive series of projects. The architecture, data models, and implementation approaches are **subject to evolution and enhancement** as subsequent CA assignments introduce additional requirements, scaling considerations, and advanced features.

**Infrastructure Optimization**: The system is specifically optimized for AWS Free Tier eligibility using t2.micro instances, which may require performance trade-offs compared to production-scale deployments.

## **HOW KAFKA WORKS & DATA FLOW EXPLANATION**

### **Apache Kafka Core Concepts**

**1. Topics & Partitions**
- **Topic**: A logical channel/category for messages (like "sensor-data")
- **Partition**: Topics are split into partitions for scalability and parallel processing
- **Offset**: Unique sequential ID for each message within a partition

**2. Producers & Consumers**
- **Producer**: Publishes messages to topics
- **Consumer**: Subscribes to topics and processes messages
- **Consumer Groups**: Multiple consumers working together to process messages

**3. Kafka Broker**
- **Broker**: Server that stores and serves messages
- **Leader/Follower**: Each partition has a leader (handles reads/writes) and followers (replicas)

**4. ZooKeeper**
- **Coordination**: Manages cluster metadata, leader election, configuration
- **Service Discovery**: Tracks which brokers are alive and healthy

### **Step-by-Step Data Flow**

**Step 1: Producer Creates Data (producer-vm: 10.0.1.12)**
```python
# simple-producer.py generates sensor data
data = {
    "sensor_id": "sensor-001",
    "temperature": 25.5,
    "humidity": 45.2,
    "pressure": 1013.25,
    "timestamp": "2025-09-07T06:15:00Z"
}
```
- Producer connects to Kafka broker at `10.0.1.162:9092`
- Serializes data to JSON bytes
- Sends to topic `sensor-data` with key `sensor-001`

**Step 2: Kafka Broker Receives & Stores (kafka-vm: 10.0.1.162)**
```
┌─────────────────────────────────────┐
│           Kafka Broker              │
│  Topic: sensor-data                 │
│  ┌─────────────────────────────────┐ │
│  │ Partition 0    │ Partition 1   │ │ 
│  │ [msg1][msg4]   │ [msg2][msg5]  │ │
│  │ [msg7]...      │ [msg8]...     │ │
│  └─────────────────────────────────┘ │
│  │ Partition 2                     │ │
│  │ [msg3][msg6][msg9]...          │ │
│  └─────────────────────────────────┘ │
└─────────────────────────────────────┘
```
- Broker receives message from producer
- Assigns message to partition (based on key hash: `sensor-001` → partition)
- Appends to partition log with unique offset number
- Acknowledges successful write to producer
- ZooKeeper tracks broker health and partition leadership

**Step 3: Consumer Polls for Messages (processor-vm: 10.0.1.245)**
```python
# simple-consumer.py polls Kafka
consumer = KafkaConsumer(
    'sensor-data',
    bootstrap_servers=['10.0.1.162:9092'],
    group_id='simple-processor'
)
```
- Consumer connects to Kafka broker
- Joins consumer group `simple-processor`
- Kafka assigns partitions to this consumer
- Consumer polls broker for new messages
- Receives messages with metadata (partition, offset)

**Step 4: Data Processing (processor-vm: 10.0.1.245)**
```python
# Consumer processes and enriches data
processed_data = {
    **data,  # Original sensor data
    'processed_at': '2025-09-07T06:15:05Z',
    'partition': message.partition,
    'offset': message.offset,
    'processed_by': 'simple-processor'
}
```
- Deserializes message from JSON
- Adds processing metadata
- Enriches with partition/offset info
- Prepares for database storage

**Step 5: MongoDB Storage (mongodb-vm: 10.0.1.160)**
```javascript
// Data stored in MongoDB
{
  "_id": ObjectId("..."),
  "sensor_id": "sensor-001",
  "temperature": 25.5,
  "humidity": 45.2,
  "pressure": 1013.25,
  "timestamp": "2025-09-07T06:15:00Z",
  "processed_at": "2025-09-07T06:15:05Z",
  "partition": 1,
  "offset": 47,
  "processed_by": "simple-processor"
}
```
- Consumer connects to MongoDB at `10.0.1.160:27017`
- Inserts processed document into `sensordata.processed_data` collection
- MongoDB returns insertion acknowledgment
- Consumer commits offset to Kafka (message successfully processed)

### **Key Benefits of This Architecture**

**Scalability:**
- **Horizontal**: Add more partitions, consumers, or brokers
- **Vertical**: Each component can be scaled independently

**Reliability:**
- **Durability**: Messages persisted to disk
- **Fault Tolerance**: ZooKeeper manages failures
- **At-Least-Once**: Messages guaranteed to be processed

**Decoupling:**
- **Producer**: Doesn't know about consumers
- **Consumer**: Can process at its own pace
- **Buffer**: Kafka acts as buffer between producer and consumer

**Real-Time Processing:**
- **Low Latency**: Messages available immediately
- **Stream Processing**: Continuous data flow
- **Event Sourcing**: Complete message history available
